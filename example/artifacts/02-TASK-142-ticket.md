# TASK-142 — Upgrade Angular frontend to v21 and adopt signals, OnPush, and built-in control flow

**Problem:** The TaskApp frontend lags behind the current Angular major and the team's standards still reference older patterns. Components rely on structural directives (*ngIf/*ngFor) and ad-hoc state, the skills documents do not mandate signals or OnPush, and the build targets an older Angular. We need to upgrade to Angular 21, migrate templates to the new @if/@for control-flow blocks, standardize on signals for state and OnPush change detection everywhere, and update the 'Add an Angular feature' skill to codify these conventions. Because this is a frontend-only change, the NestJS backend contract that the frontend consumes must remain byte-for-byte unchanged so the upgraded UI keeps working; that contract stability is what is verifiable by Jest.

**In scope:** Upgrade Angular workspace and dependencies to v21 for apps/taskapp/frontend; Migrate all templates from *ngIf/*ngFor structural directives to @if/@for/@switch control-flow blocks; Convert component state to signal()/computed() and set ChangeDetectionStrategy.OnPush on every component; Update the 'Add an Angular feature' skill and CODING_STANDARDS to require signals, OnPush, and control-flow blocks; Confirm the existing NestJS task API contract is unchanged so the upgraded frontend continues to function
**Out of scope:** Any change to backend modules, controllers, services, DTOs, or routes; Adding new product features or task fields; Backend storage or migration changes (storage remains in-memory); Changes to shared-types contracts

## Changes
- **Backend:** No production code changes; backend is verified to remain unchanged
- **Frontend:** Bump Angular and Angular CLI to v21 in apps/taskapp/frontend package.json and run the official ng update migrations, Replace *ngIf/*ngFor with @if/@for (with track) and @switch across all feature components, Set ChangeDetectionStrategy.OnPush on every standalone component, Replace BehaviorSubject/mutable fields with signal()/computed() for view state, Keep all HttpClient access inside typed core/api services rendering dates via TzDatePipe
- **Shared:** No changes to libs/shared-types contracts

## Acceptance criteria
- **AC-1** — Given the upgraded frontend posts a valid task payload to the backend, when a Jest e2e test sends POST /tasks with a title via supertest, then the backend responds 201 with a task object containing an id, the title, completed=false, and a createdAt that is a valid UTC ISO-8601 string, confirming the create contract is unchanged.
- **AC-2** — Given tasks exist in the backend, when a Jest e2e test sends GET /tasks, then the backend responds 200 with a JSON array whose items match the existing Task shape (id, title, completed, UTC ISO date fields), confirming the list contract the frontend depends on is unchanged.
- **AC-3** — Given a task has been created, when a Jest e2e test sends PATCH /tasks/:id/complete for that id, then the backend responds 200 and the returned task has completed=true, confirming the complete-task contract is unchanged.
- **AC-4** — Given a task has been created, when a Jest e2e test sends DELETE /tasks/:id and then GET /tasks/:id, then the delete responds 200 with {ok:true} and the subsequent fetch responds 404, confirming the delete contract is unchanged.
- **AC-5** — Given a request that violates the CreateTaskDto validation rules, when a Jest e2e test sends POST /tasks with an invalid body (e.g. missing/empty title), then the backend responds with a 400 HttpException and does not create a task, confirming input validation behavior is unchanged.
