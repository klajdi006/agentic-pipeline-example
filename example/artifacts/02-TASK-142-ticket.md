# TASK-142 — Task Priorities: complete CRUD across backend and frontend

**Problem:** A partial task-priority backend was introduced in an earlier commit but the update path is incomplete and the frontend has drifted out of contract, leaving no working CRUD surface. Users cannot create, view, edit, or delete task priorities end-to-end, and shared request/response contracts are inconsistent between FE and BE. We need to finish the backend update path, build the full Angular frontend surface, align shared types, and close the test-tooling gaps so the feature is complete and verified.

**In scope:** Finish backend Priorities CRUD: complete the Update path and ensure Create, Read (list + single), Update, and Delete are all implemented in the NestJS priorities module; DTO validation with class-validator for create and update inputs; Align libs/shared-types contracts for priority request/response and import them on both FE and BE; Build Angular standalone OnPush priorities feature: list, create, edit, delete views using signals and a typed core/api service; Jest unit tests for PrioritiesService and e2e (*.e2e-spec.ts) tests for PrioritiesController; Frontend component tests with Testing Library covering the acceptance criteria
**Out of scope:** Persistent database storage / TypeORM migrations (storage remains in-memory); Assigning priorities to tasks or reordering logic beyond basic CRUD; Authentication/authorization changes; Reminder/scheduler changes

## Changes
- **Backend:** Complete src/priorities/ module: controller (thin, one route per use case), service (business logic), and dto/ (CreatePriorityDto, UpdatePriorityDto with class-validator), Implement the missing Update path and confirm Create/list/get/Delete behavior, Throw HttpException subclasses (e.g. NotFoundException) for missing resources; never return raw error strings, Return UTC ISO-8601 timestamps where applicable, Add priorities.service.spec.ts (unit) and priorities.e2e-spec.ts (controller)
- **Frontend:** Add standalone OnPush feature components under src/app/features/priorities/ for list, create, and edit, Add a typed core/api/priorities.service.ts returning shared-types contracts (no direct HttpClient in components), Use signal()/computed() for view state and typed reactive forms mirroring DTO validation, Add a lazy loadComponent route and a nav entry, Render any date fields via TzDatePipe, Add Testing Library component tests covering CRUD acceptance criteria
- **Shared:** Define/align Priority, CreatePriorityRequest, UpdatePriorityRequest, and PriorityResponse contracts in libs/shared-types/ and import them in both FE service and BE DTOs/controller

## Acceptance criteria
- **AC1** — Given an empty priorities repository, when a POST request is sent to the priorities endpoint with a valid create payload, then the API responds 201 with the created priority including a generated id and the submitted fields.
- **AC2** — Given a create request with an invalid or missing required field, when the POST request is processed, then the API responds 400 and the priority is not persisted.
- **AC3** — Given existing priorities in the repository, when a GET request is sent to the priorities list endpoint, then the API responds 200 with an array containing all stored priorities.
- **AC4** — Given an existing priority id, when a GET request is sent for that single priority id, then the API responds 200 with the matching priority.
- **AC5** — Given a priority id that does not exist, when a GET, PATCH/PUT, or DELETE request targets that id, then the API responds 404 via an HttpException without persisting changes.
- **AC6** — Given an existing priority, when a PATCH/PUT request is sent with a valid update payload, then the API responds 200 with the updated fields reflected and unchanged fields preserved.
- **AC7** — Given an update request with an invalid field value, when the update request is processed, then the API responds 400 and the stored priority is left unchanged.
- **AC8** — Given an existing priority, when a DELETE request is sent for that id, then the API responds with a success status and a subsequent GET for that id responds 404.
