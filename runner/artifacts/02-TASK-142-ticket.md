# TASK-142 — Server-side paginated task list

**Problem:** The GET /tasks endpoint returns the full unbounded array on every request and the frontend has no pagination UI. As the task count grows this degrades both API performance and usability. The backend must slice results server-side and return metadata (total, page, limit) so the frontend can render prev/next controls and page through tasks efficiently.

**In scope:** Accept page (default 1, min 1) and limit (default 20, min 1, max 100) query params on GET /tasks; Return PaginatedResponse { items, total, page, limit } from GET /tasks instead of a bare array; Add PaginationQueryDto with class-validator decorators to validate and default both params; Add generic PaginatedResponse<T> interface to libs/shared-types (reusable for future resources); Add TaskResponse interface to libs/shared-types and export PaginatedTasksResponse = PaginatedResponse<TaskResponse>; Unify FE and BE Task models via shared-types — FE model gains the priority field; Update TasksService.findAll to sort then slice the in-memory store and return { items, total, page, limit }; Refactor TaskListComponent to use httpResource() driven by page and pageSize signals instead of ngOnInit + subscribe; Add prev/next pagination controls and loading state to task-list.component.html; Persist current page and limit in the URL as query params (?page=2&limit=20) using withComponentInputBinding(); Add Jest unit tests to tasks.service.spec.ts covering paginated slicing, empty pages, and out-of-bound pages; Add Jest e2e tests to tasks.controller.e2e-spec.ts covering default params, custom params, edge pages, and invalid inputs
**Out of scope:** sortBy / sortDir query parameters — sort remains locked to existing priority-rank order; Infinite scroll or numbered page-button UI (prev/next only); TypeORM or database-level pagination (in-memory store only for now); Moving the task list to a dedicated /tasks route — component stays embedded in AppComponent; Horizontal-scale advisory lock for the scheduler (separate concern tracked in ADR-0001 follow-up)

## Acceptance criteria
- **AC-1** — Given the in-memory task store contains 25 tasks, when GET /tasks is called with no query parameters, then the response status is 200 and the body matches { items: <array of 20 tasks>, total: 25, page: 1, limit: 20 } with items.length === 20.
- **AC-2** — Given the in-memory task store contains 25 tasks, when GET /tasks?page=2&limit=10 is called, then the response status is 200 and the body matches { items: <array of 10 tasks>, total: 25, page: 2, limit: 10 } where items are tasks at sorted positions 11–20.
- **AC-3** — Given the in-memory task store contains 25 tasks, when GET /tasks?page=3&limit=10 is called, then the response status is 200 and body.items.length === 5 and body.total === 25 and body.page === 3.
- **AC-4** — Given the in-memory task store contains 25 tasks, when GET /tasks?page=4&limit=10 is called, then the response status is 200 and body.items is an empty array and body.total === 25.
- **AC-5** — Given the in-memory task store is empty, when GET /tasks is called with no query parameters, then the response status is 200 and the body matches { items: [], total: 0, page: 1, limit: 20 }.
- **AC-6** — Given any task store state, when GET /tasks?limit=101 is called, then the response status is 400 and the error body references the limit field.
- **AC-7** — Given any task store state, when GET /tasks?page=0 is called, then the response status is 400 and the error body references the page field.
- **AC-8** — Given any task store state, when GET /tasks?limit=0 is called, then the response status is 400 and the error body references the limit field.
- **AC-9** — Given the in-memory task store contains tasks with mixed priority ranks, when GET /tasks?page=1&limit=5 is called, then the response status is 200 and body.items are ordered by ascending priority rank matching the full store sort order.
- **AC-10** — Given the in-memory task store contains 5 tasks, when GET /tasks?page=1&limit=100 is called, then the response status is 200 and body.items.length === 5 and body.total === 5 and body.limit === 100.
